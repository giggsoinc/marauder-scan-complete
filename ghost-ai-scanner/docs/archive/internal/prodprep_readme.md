## Overview

This document captures every task required to move PatronAI from demo (single EC2 + docker-compose) to production (AWS Fargate + Terraform). Read this before touching any production infrastructure.

> **On naming:** Infrastructure identifiers (env vars, S3 bucket prefix,
> IAM user name, SNS topic, Parameter Store keys) keep the `marauder-scan`
> token to preserve the v1 deployment and avoid breaking VPC Flow Log
> targets, CloudTrail filters and IAM policies. The **product name** is
> PatronAI everywhere a human reads it.

Two phases clearly separated:

**Phase 1 — Demo (this week)**  
Single EC2. docker-compose. Real AWS S3 and SNS. setup.sh handles prerequisites. One command to run.

**Phase 2 — Production (next week post-demo)**  
AWS Fargate. Two ECS services. ALB routing. Terraform IaC. OAuth via Streamlit native auth or ALB + Cognito. Auto-scaling Grafana.

## PHASE 1 — Demo Prerequisites

### What setup.sh handles automatically

Run `scripts/setup.sh` once before `docker-compose up`. It does the following:

| # | Task | AWS CLI command used |
|---|---|---|
| 1 | Create S3 bucket | `aws s3 mb` |
| 2 | Enable S3 versioning | `aws s3api put-bucket-versioning` |
| 3 | Set S3 lifecycle rules — 30 day hot, archive after | `aws s3api put-bucket-lifecycle-configuration` |
| 4 | Block all public S3 access | `aws s3api put-public-access-block` |
| 5 | Create IAM user `marauder-scan` *(v1 infra name — PatronAI reuses it)* | `aws iam create-user` |
| 6 | Create and attach IAM policy | `aws iam put-user-policy` |
| 7 | Generate IAM access key | `aws iam create-access-key` |
| 8 | Create SNS topic `marauder-scan-alerts` | `aws sns create-topic` |
| 9 | Subscribe alert email to SNS | `aws sns subscribe` |
| 10 | Enable VPC Flow Logs on all VPCs | `aws ec2 create-flow-logs` |
| 11 | Seed S3 config files | `aws s3 cp` — settings.json, authorized.csv, unauthorized.csv |
| 12 | Generate .env file | Written locally |
| 13 | Generate packetbeat.yml | Written locally with domain filters |
| 14 | Generate agent/config.json | Written locally with bucket and region |
| 15 | Generate grafana/datasources/s3.json | Written locally pointing at bucket |

### What IT Admin does manually before setup.sh

- AWS CLI installed and configured with admin credentials (`aws configure`)
- Docker and docker-compose installed on EC2
- EC2 security group: port 8501 open to office IP only, port 3000 open to office IP only, port 80 open to office IP only
- Git clone of the PatronAI repo onto EC2 (legacy `marauder-scan` URL still redirects)

## PHASE 2 — Production Migration Tasks

### 2.1 Terraform — what to build

All Terraform lives in `terraform/` directory. One module per concern.

| File | What it builds | Priority |
|---|---|---|
| `terraform/aws/main.tf` | Provider config, backend (S3 state), module calls | P1 |
| `terraform/aws/s3.tf` | S3 bucket, versioning, lifecycle, bucket policy, access logging | P1 |
| `terraform/aws/ecr.tf` | Two ECR repos: `patronai` and `patronai-grafana` (with legacy `marauder-scan` aliases) | P1 |
| `terraform/aws/iam.tf` | Task execution role, scanner task role, Grafana task role | P1 |
| `terraform/aws/ecs.tf` | Two ECS services: scanner (fixed 1) and Grafana (auto-scale 1-5) | P1 |
| `terraform/aws/alb.tf` | Internal ALB, two target groups, path routing /settings and /grafana | P1 |
| `terraform/aws/variables.tf` | All input variables — company_name, vpc_id, subnet_ids etc | P1 |
| `terraform/aws/outputs.tf` | Grafana URL, Streamlit URL, S3 bucket, ECR URLs | P1 |
| `terraform/edge/kinesis.tf` | Firehose delivery stream — Packetbeat to S3 | P2 |
| `terraform/edge/iam_edge.tf` | IAM role for edge agents and Packetbeat | P2 |
| `terraform/gcp/` | GCS, Cloud Run, IAM stubs | P3 |
| `terraform/azure/` | Blob, ACI, IAM stubs | P3 |

### 2.2 Replace IAM access keys with instance profile

Demo uses access keys in .env. Production uses IAM instance profile attached to the ECS task.

**Action:** Remove `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` from .env and docker-compose. Attach the IAM role to the Fargate task definition instead. The scanner uses boto3 which auto-discovers instance credentials.

```hcl
# In terraform/aws/ecs.tf
resource "aws_ecs_task_definition" "scanner" {
  task_role_arn      = aws_iam_role.scanner_task.arn
  execution_role_arn = aws_iam_role.ecs_execution.arn
  ...
}
```

### 2.3 Replace .env with AWS Systems Manager Parameter Store

Demo uses .env file on EC2. Production stores all secrets in Parameter Store. Container reads at startup.

| .env variable | Parameter Store key |
|---|---|
| MARAUDER_SCAN_BUCKET | /marauder-scan/{company}/bucket |
| ALERT_SNS_ARN | /marauder-scan/{company}/sns-arn |
| TRINITY_WEBHOOK_URL | /marauder-scan/{company}/trinity-webhook |
| GF_SECURITY_ADMIN_PASSWORD | /marauder-scan/{company}/grafana-password |
| ALLOWED_EMAILS | /marauder-scan/{company}/allowed-emails |

Non-sensitive config (region, scan interval) stays as ECS task environment variables.

### 2.4 Replace email allowlist with OAuth

Demo uses email text input allowlist. Production uses proper OAuth.

**Option A — Streamlit native auth (simplest)**

```toml
# .streamlit/config.toml
[auth]
redirect_uri = "https://settings.security.internal/oauth2/callback"
cookie_secret = "from-parameter-store"

[auth.microsoft]
client_id     = "from-azure-app-registration"
client_secret = "from-parameter-store"
tenant_id     = "customer-tenant-id"
```

Requires Streamlit >= 1.42. Customer needs Azure App Registration (30 min setup).

**Option B — ALB + Cognito (enterprise grade)**

ALB handles OAuth before traffic reaches Streamlit. Zero code change in app.py. Terraform builds the Cognito User Pool and ALB authentication action.

```hcl
# terraform/aws/alb.tf addition
resource "aws_alb_listener_rule" "auth" {
  action {
    type = "authenticate-cognito"
    authenticate_cognito {
      user_pool_arn       = aws_cognito_user_pool.main.arn
      user_pool_client_id = aws_cognito_user_pool_client.main.id
      user_pool_domain    = aws_cognito_user_pool_domain.main.domain
    }
  }
}
```

**Recommendation:** Option A for first production release. Option B for enterprise customers.

### 2.5 Enable Network Firewall logging

Customer must run before Terraform deploy:

```bash
./deploy/enable_firewall_logs.sh marauder-scan-{company} us-east-1
```

This enables AWS Network Firewall DNS and connection logging and configures delivery to the OCSF S3 prefix.

### 2.6 Deploy OCSF normalisation Lambda

Customer must run:

```bash
terraform -chdir=deploy/ init
terraform -chdir=deploy/ apply
```

This deploys:
- Firehose transform Lambda (`ocsf_transform.tf`)
- VPC Flow Log enablement (`enable_flowlogs.sh`)
- S3 bucket policies for log delivery

### 2.7 Multi-cloud migration notes

When moving to GCP or Azure set `CLOUD_PROVIDER` env var. The providers/ adapter layer handles the rest. Core scanner code unchanged.

| AWS service | GCP equivalent | Azure equivalent | Lock-in level |
|---|---|---|---|
| S3 | GCS | Blob Storage | MEDIUM |
| SNS | Pub/Sub | Service Bus | MEDIUM |
| Fargate | Cloud Run | ACI | MEDIUM |
| VPC Flow Logs | VPC Flow Logs (different format) | NSG Flow Logs | HIGH |
| IAM Identity Center | Cloud Identity | Entra ID | HIGH |
| CloudTrail | Cloud Audit Logs | Activity Log | HIGH |
| Parameter Store | Secret Manager | Key Vault | MEDIUM |
| ECR | Artifact Registry | ACR | LOW |

HIGH lock-in requires code change in `providers/` adapter only. Core scanner unchanged.

### 2.8 Packetbeat domain filter update process

When `unauthorized.csv` is updated with new providers, Packetbeat filter config must be regenerated and redeployed to all managed devices.

**Process:**
1. Giggso updates `config/unauthorized.csv` in S3
2. Run `scripts/generate_packetbeat_config.sh` to regenerate `packetbeat.yml`
3. Push updated config via MDM (Jamf/Intune/Ansible)
4. Packetbeat restarts on each device and picks up new filters

Target: complete within 24 hours of CSV update.

### 2.9 Grafana production hardening

| Task | Action |
|---|---|
| Change default admin password | Set `GF_SECURITY_ADMIN_PASSWORD` from Parameter Store |
| Disable Grafana anonymous access | `GF_AUTH_ANONYMOUS_ENABLED=false` |
| Enable HTTPS | Terminate TLS at ALB, Grafana runs HTTP internally |
| Restrict Grafana data source | S3 datasource read-only IAM policy |
| Enable audit logging | `GF_LOG_MODE=console file` |
| Set session timeout | `GF_SESSION_LIFE_TIME=86400` |

### 2.10 S3 storage scaling thresholds

At 10,000 managed devices the following S3 costs and sizes apply per year (with Packetbeat domain filter enabled):

| Path | Per day | Per year | Action at threshold |
|---|---|---|---|
| `cursor/state.json` | 500 bytes | 500 bytes | Never grows |
| `findings/` | 400KB | 146MB | Archive after 30 days |
| `dedup/` | 160KB | 58MB | Delete after 90 days |
| `summary/` | 15KB | 5.5MB | Keep 365 days |
| `ocsf/` filtered | 2.5MB | 912MB | Archive after 30 days |
| `reports/` | 500KB | 182MB | Keep indefinitely |

When findings exceed 10,000 events per day consider moving to Athena queries instead of S3 Select. Document this decision in the architecture review at 6 months.

### 2.12 Monitoring the scanner itself

The scanner must be monitored. Add these CloudWatch alarms in production:

| Alarm | Condition | Action |
|---|---|---|
| Scanner lag | Cursor not updated in 15 min | SNS to Platform Team |
| High memory | Container memory > 80% | Scale up or investigate |
| S3 write failures | S3 put errors > 5 in 5 min | SNS to Platform Owner |
| No findings written | 0 findings for 60 min during business hours | SNS to Platform Owner |
| Grafana down | Health check failing | ECS restarts automatically |
