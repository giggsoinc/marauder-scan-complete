What's New in v1.0.0

SEC-01

PatronAI v1.0.0 is the first generally-available release. It ships three role-based views, a full reports module, an AI chat widget, and a pluggable LLM layer — all deployable to a single EC2 instance in under 30 minutes using the included automated deploy script.

Executive View

  • AI security KPIs at a glance
  • Clickable drill-down metrics
  • Landscape, Risk, and Exposure tabs
  • AI Chat Widget integrated

Manager View

  • Inventory, Risks, Logs, Pipeline tabs
  • AI Inventory: MCP / agent / vector DB mind map
  • Per-user picker on mind map
  • AI Chat Widget integrated

Support View

  • Rules, Code Signals, Coverage, Health
  • Logs, Risks, Pipeline, Agent Fleet tabs
  • Agent fleet status at a glance
  • AI Chat Widget integrated

User Detail Page

  • Per-user AI risk profile
  • AI asset mind map (scoped to user)
  • Full findings drill-through

Provider Lists

  • Editable allow-list per AI provider
  • Block-list with enforcement flags
  • Changes persisted to S3

Users & Branding

  • Add / remove users, role assignment
  • Welcome email on user add
  • Logo upload to S3 (embedded in PDFs)
Deploy script included. Full automated deploy covers code transfer, .env builder, MCP setup, LLM service setup (systemd), and prints the Claude Desktop config — target host is up in < 30 minutes.

Reports Module — R1 through R7

ID Report Name Description Formats
R1 Executive Risk Summary High-level KPIs: risk posture, top findings, trend lines. Designed for board or C-suite distribution. HTMLPDF

AI Chat Widget & MCP Server

SEC-03
  • Chat Widget — Ask AI button available in Executive, Manager, and Support views. Collapsed by default; click to expand inline.
  • 8 built-in tools exposed to the chat agent: query findings, look up user risk, list assets, check fleet health, pull report data, search providers, audit trail query, and coverage check.
  • Conversation history persisted to S3 under a SHA-256(email)-keyed prefix — no email address ever appears in the S3 object path.
  • MCP Server (stdio transport) exposes PatronAI as a tool to Claude Desktop and Cursor IDE. Access requires SSH key to the EC2 host.
  • Pluggable LLM backend via openai_compat adapter — supports llama.cpp, Ollama, OpenAI, and Groq interchangeably. Anthropic client also supported. Backend selected via .env or AWS Parameter Store at startup.
  • Primary model: gpt-4.1-mini  |  Fallback: gpt-4.1-mini (configurable). All LLM calls use temperature=0.

Security Model & Architecture

SEC-04

PatronAI v1.0.0 is a single-tenant deployment on a dedicated EC2 instance. All persistent state lives in a single S3 bucket (patronai). There is no Oracle dependency.

Identity & Users

User records stored in s3://patronai/users/users.json. Role assignment (exec / manager / support) enforced at view render. No external IdP dependency in v1.0.

Credentials

All secrets loaded from .env at runtime. No credentials are hardcoded in source. AWS access via named profile (rvdts) on the EC2 host.

MCP Access Control

MCP server uses stdio transport. External access requires an SSH key to the EC2 host. No unauthenticated HTTP endpoint is exposed for MCP in v1.0.

Chat History Privacy

History keyed by SHA-256(email). Email is never stored as a plaintext S3 path component. Bucket-level IAM is the boundary in v1.0 (see RISK-4).

Audit Integrity

R6 Compliance Audit Trail records carry SHA-256 hashes. Report generation timestamps are immutable once written to S3.

Transport

All external traffic served via nginx reverse proxy on the EC2 host. TLS termination handled at the nginx layer.

Known Risk Gaps — Targeted for V2

SEC-05

The following security gaps are documented and accepted for v1.0.0. Each has a defined mitigation strategy targeted for V2. No gap is a blocker for initial deployment in a controlled enterprise environment.

Risk ID Severity Description V2 Mitigation Target
RISK-1 HIGH Overly-broad IAM role on MCP server.
The MCP server process inherits the EC2 instance IAM role, which has broader than read-only S3 access. A compromised MCP process could read or write outside the patronai bucket. 		Dedicated IAM role Least-privilege S3
Create a dedicated IAM role scoped to s3:GetObject / s3:PutObject on patronai/* only. Attach via instance profile swap or task role. 		V2.0
RISK-2 MEDIUM MCP access requires SSH — not suitable for non-admin users.
The stdio MCP transport means only users with SSH key access to the EC2 host can use Claude Desktop / Cursor integration. This excludes non-technical analysts. 		SSE transport JWT RS256 auth
Migrate MCP server to SSE transport with an HTTP endpoint. Issue short-lived JWT tokens (RS256) per authenticated user. No SSH required. 		V2.0
RISK-3 LOW No per-call rate limiting on MCP tools.
Rapid successive MCP tool calls incur S3 GET/LIST costs and could degrade service availability. There is no per-key or per-user throttle in v1.0. 		Per-key rate limiting
Implement token-bucket rate limiting per API key / SSH identity at the MCP server layer. Configurable threshold via environment variable. 		V2.0
RISK-4 INFO Chat history visible to any principal with bucket access.
Chat history objects are stored under a SHA-256(email) prefix. Email is not in the path, but any IAM principal with s3:GetObject on patronai/* can read all users' history. 		Per-user KMS key
Encrypt each user's history prefix with a dedicated AWS KMS key. IAM key policies enforce per-user decryption rights; bucket access alone is insufficient. 		V2.0
Note for operators: RISK-1 (HIGH) should be remediated before exposing the MCP server to users outside the immediate admin team. The remaining risks are acceptable in a controlled internal deployment.

Deferred to v1.1

SEC-06

The following features were scoped for v1.0 but deferred to v1.1 to keep the initial release focused and shippable. Each item has a full design memo.

V1.1
Email / SMTP Module

Per-user outbound email via configurable SMTP. Credentials stored in AWS Secrets Manager (never in .env). Includes a test-send-before-save flow and per-user credential isolation. Welcome-email on user-add is triggered from this module. Designed as a pluggable optional module (~965 LOC estimated). SES-based approach was evaluated and superseded by this SMTP design.

Current behavior: The welcome email on user-add in v1.0 is a stub — the UI shows a confirmation but no email is actually sent until the v1.1 SMTP module ships.
