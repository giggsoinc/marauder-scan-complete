1 Overview

PatronAI's AI layer adds two distinct entry points for LLM-powered interaction with the security scan data already held in S3. Both entry points share the same 8 pure analytics tools and the same data layer — no duplicate business logic exists.

Chat Widget (Streamlit)

  • Embedded in Exec, Manager, and Support role views
  • Collapsed expander — minimal footprint on the dashboard
  • Per-user, per-view conversation history persisted to S3 JSONL
  • Transport-agnostic engine — swap LLM by changing one env var
  • V1 default: local llama-server (zero cloud cost)

MCP Server (Claude Desktop / Cursor)

  • Exposes the same 8 tools over MCP stdio protocol
  • No network port in V1 — SSH RSA enforces access at the OS level
  • Reads live S3 data on each tool call via BlobIndexStore
  • No separate auth code required — authorized_keys is the gatekeeper
  • Designed for admin/analyst use via Claude Desktop or Cursor IDE
Design Principle Tools in src/chat/tools.py are pure functions — no LLM calls, no I/O, no side effects. Both the Streamlit chat engine and the MCP server call the same functions. The LLM is always outside the data layer.

Module Responsibilities

Module Responsibility External I/O
tools.py 8 analytics functions over in-memory event data: severity breakdowns, trend windows, asset rankings, alert queues, etc. None — pure Python
tools_schema.py OpenAI JSON schema dict list describing each tool's name, description, and parameters. Used by both engine.py and the MCP server. None
prompts.py Builds the system prompt at request time. Injects the caller's role view (exec/manager/support) and a live data snapshot so the LLM has grounding context without a tool call. Reads live events in-memory
history.py Reads and appends conversation turns to S3 JSONL. Path pattern: chat/{sha256(email)[:16]}/{view}/YYYY-MM-DD.jsonl. Hashed email prevents PII exposure in S3 key paths. AWS S3 (patronai bucket)
engine.py Orchestrates the multi-turn tool-call loop. Injects system prompt, calls client.complete(), executes any tool calls from tools.py, feeds results back, repeats until the LLM returns plain text. Calls LLM via llm/ client
widget.py Streamlit UI — renders a collapsed expander with a message input, displays conversation history, calls engine.py, then calls history.py to persist the turn. Streamlit session state
llm/__init__.py Factory function get_client() — reads LLM_PROVIDER from env, falls back to AWS SSM Parameter Store path /patronai/llm/provider, returns the correct LLMClient subclass. Env / AWS SSM
llm/base.py Abstract base class LLMClient. Defines the contract: complete(messages, tools) and tool_result_msg(tool_call_id, result). All concrete clients must implement both. None
llm/openai_compat.py Concrete client for any OpenAI-compatible endpoint — llama.cpp, Ollama, OpenAI, Groq. Sends POST /v1/chat/completions with the OpenAI tool schema format. HTTP to configured base URL
llm/anthropic.py Concrete client for Anthropic Messages API. Converts OpenAI-format tool schemas to Anthropic format internally so engine.py stays provider-agnostic. HTTPS to api.anthropic.com

3 LLM Transport

V1 Default — Local llama.cpp (Zero Cloud Cost) The V1 default runs entirely on EC2. No API keys are needed. The llama-server process is started separately and exposes an OpenAI-compatible endpoint at http://localhost:8080. No .env changes are required to run with the default local model.

V1 — llama.cpp (Default)

  • Process: llama-server on EC2
  • Endpoint: http://localhost:8080/v1/chat/completions
  • Model: Qwen 3.5 9B Q4_K_M GGUF
  • Provider constant: openai_compat
  • Zero config — defaults are built into llm/__init__.py
  • No outbound HTTPS calls, no API billing

Cloud Override (any provider)

  • Set LLM_PROVIDER, LLM_BASE_URL, LLM_API_KEY, LLM_MODEL
  • Via .env file or AWS Parameter Store /patronai/llm/*
  • Works with OpenAI, Groq, Ollama, Anthropic
  • SSM takes precedence over local env for cloud deployments
  • No code change required — only config

Provider Resolution Order

# llm/__init__.py — get_client() resolution chain

1. Read LLM_PROVIDER from os.environ
2. If not set → query SSM /patronai/llm/provider
3. If still not set → default to "openai_compat"

# Same chain for each config key:
LLM_BASE_URL   → SSM /patronai/llm/base_url   → "http://localhost:8080"
LLM_MODEL      → SSM /patronai/llm/model      → "qwen3.5-9b-q4"
LLM_API_KEY    → SSM /patronai/llm/api_key    → ""  (empty — not needed for local)

Data Loading on Each Call

# patronai_mcp_server.py — data lifecycle

# On each tool call, the server loads fresh data from S3:
store = BlobIndexStore(bucket="patronai", aws_profile="rvdts")
events = store.load_events()          # pulls latest index + event blobs

# Then delegates to the same tools.py function:
result = tools.get_severity_breakdown(events)
# No state held between calls — always reflects live S3 data
V2 Planned — SSE/HTTP Transport V2 will add an HTTP+SSE transport option so non-SSH clients (web IDEs, internal tooling) can reach the MCP server without an SSH tunnel. This requires a dedicated network port, JWT RS256 token validation, and a least-privilege IAM role. See the Security Model section for the V2 design.

5 Security Model

V1 — Current (SSH stdio)
  • Transport: stdio via SSH — no open network port on the host
  • Auth: OS-enforced RSA key pair — no application-level auth code
  • Access control: ~/.ssh/authorized_keys on EC2
  • Admin adds/removes public keys to grant or revoke access
  • Each connection is a separate subprocess — no shared state
  • IAM: uses the EC2 instance role (existing rvdts profile permissions)
  • S3 data: read-only access — MCP server never writes to S3
  • No secrets in code — all config via .env or SSM
V2 — Planned (SSE/HTTP)
  • Transport: HTTP + Server-Sent Events on a dedicated port
  • Auth: JWT RS256 tokens (short-lived, signed by PatronAI auth service)
  • Dedicated IAM role: least-privilege — s3:GetObject on patronai/events/* only
  • Token validation middleware in patronai_mcp_server.py
  • Rate limiting per token identity
  • Audit log: every tool call logged with token subject + timestamp
  • Nginx reverse proxy handles TLS termination
V2 Risk: Network Exposure Enabling the HTTP+SSE transport opens a new network port. Before enabling V2, confirm: (1) the EC2 security group restricts inbound to known CIDR ranges, (2) the IAM role is scoped to s3:GetObject on patronai/events/* only — not the full bucket, (3) JWT signing keys are stored in AWS Secrets Manager, not in .env.
V2 Risk: IAM Scope Creep The V1 MCP server inherits the full EC2 instance role. In V2, a dedicated IAM role with explicit deny on all non-event S3 paths must be created before the HTTP transport is enabled. The current instance role has broader S3 permissions that are appropriate for the Streamlit app but excessive for the MCP server alone.

Chat Widget — Identity Model

User identity for conversation history scoping comes from the Streamlit session. The email is read from the session (authenticated via PatronAI's existing login flow), SHA-256 hashed, and the first 16 hex characters are used as the S3 key prefix. This avoids storing PII in S3 key paths while still providing per-user isolation. 

# history.py — identity hashing
import hashlib

def user_prefix(email: str) -> str:
    return hashlib.sha256(email.encode()).hexdigest()[:16]

# "admin@example.com" → "a3f9b2c1d4e5f6a7"
# Used only as S3 key prefix — never stored as user record

6 Data Flow

Chat Widget — Request Lifecycle

1
widget.py — User types message into Streamlit input. widget.py collects the text and retrieves existing history from S3 via history.py (or starts a new empty list).
│
2
engine.py — Receives message list. Calls prompts.py to build a view-scoped system prompt with a live data snapshot. Prepends system prompt to message list.
│
3
llm/openai_compat.py — engine.py calls client.complete(messages, tools). Client POSTs to llama-server at http://localhost:8080/v1/chat/completions with the OpenAI tool schema. temperature=0.
│
4
llama-server (Qwen 3.5 9B) — Returns a response. If it contains tool_calls, engine.py extracts the tool name and arguments.
│
5
tools.py — engine.py dispatches to the named function (e.g. get_severity_breakdown). Function runs against in-memory events. Returns a plain Python dict.
│
6
engine.py (loop) — Appends the tool result to the message list as a tool_result message. Calls client.complete() again. Repeats steps 4-6 until the LLM response contains no tool_calls.
│
7
widget.py — Receives final plain-text answer from engine.py. Displays it in the Streamlit chat UI.
│
8
history.py — Appends the new user turn and assistant turn to the S3 JSONL file for this user+view+date. Next page load will pre-load this history.

MCP Server — Tool Call Lifecycle

# Simplified MCP server call flow

Claude Desktop  ──stdio──►  patronai_mcp_server.py
                            │
                            ├─ BlobIndexStore.load_events()  ──►  S3 patronai/events/*
                            │   (fresh load on each call)
                            │
                            ├─ tools.get_severity_breakdown(events)
                            │   (pure function, no LLM)
                            │
                            └──► returns JSON result  ──stdio──►  Claude Desktop
No LLM in the MCP Server The MCP server is data-only. It never calls an LLM. The LLM reasoning happens inside Claude Desktop or Cursor — the MCP server just provides structured data in response to tool calls. This keeps the server stateless and fast.

7 Configuration Reference

All configuration is read from environment variables (set in .env or the shell). For EC2 deployments, any key can alternatively be stored in AWS Parameter Store under the /patronai/llm/ prefix — SSM is checked when the env var is absent.

Variable Default Required Description
LLM_PROVIDER openai_compat OPTIONAL LLM client to use. Values: openai_compat (llama.cpp / Ollama / OpenAI / Groq) or anthropic.
LLM_BASE_URL http://localhost:8080 OPTIONAL Base URL for the OpenAI-compatible API. Points to local llama-server by default. Override for cloud providers (e.g. https://api.openai.com).
LLM_MODEL qwen3.5-9b-q4 OPTIONAL Model identifier passed in the model field of every API request. Ignored by llama-server (server determines model at startup), but required for cloud endpoints.
LLM_API_KEY (empty string) OPTIONAL API key sent as Authorization: Bearer <key>. Not needed for local llama-server. Required for OpenAI, Anthropic, Groq, etc.
AWS_PROFILE rvdts OPTIONAL AWS credentials profile for S3 and SSM access. On EC2 with an instance role, the instance role is used automatically — this variable is only relevant for local development.
PATRONAI_BUCKET patronai OPTIONAL S3 bucket name for all PatronAI data (events, chat history, user registry). Single-tenant — do not share with other products.
LLM_TEMPERATURE 0 OPTIONAL Sampling temperature for all LLM calls. Hardcoded to 0 per project policy (deterministic, reproducible responses). Override is available but discouraged.
LLM_MAX_TOKENS 2048 OPTIONAL Maximum token count in the LLM response. Increase for verbose tool-result summarisation; decrease to reduce latency on lightweight queries.
