Phase 6 · Per developer
6
Deploy hook agents to developers
6.1Generate and send an installer (Admin)
BROWSER · Settings → Deploy Agents
  1. Go to Settings → Deploy Agents tab
  2. Fill in the developer's Name, Email, and Platform (Mac / Linux / Windows)
  3. Click Generate & Send
  4. Developer receives an email with a 48-hour download link and a 6-digit OTP
  5. Status column shows PENDING → INSTALLED once they run it
6.2Developer installs on Mac or Linux
DEVELOPER MACHINE
Developer pastes the link from the email:
curl -fsSL "<link-from-email>" | bash
Then enters their 6-digit OTP when prompted. That's it — hook is installed, heartbeat starts.
6.3Developer installs on Windows
DEVELOPER MACHINE · PowerShell (Admin) 
Invoke-Expression (Invoke-WebRequest -Uri "<link-from-email>").Content
6.4Build macOS DMG (optional — run on your Mac, not EC2)
YOUR MAC
Replace <token> with the token from the Deploy Agents table and <bucket> with your S3 bucket name.
# Download the rendered installer
aws s3 cp s3://<bucket>/config/HOOK_AGENTS/<token>/setup_agent.sh /tmp/setup_agent.sh

# Build DMG (requires macOS)
bash ghost-ai-scanner/build/build_dmg.sh \
  --input /tmp/setup_agent.sh \
  --name "Alice" \
  --token <token> \
  --output /tmp/PatronAI-Agent-Alice.dmg

# Upload back to S3
aws s3 cp /tmp/PatronAI-Agent-Alice.dmg \
  s3://<bucket>/config/HOOK_AGENTS/<token>/PatronAI-Agent-Alice.dmg

Phase 1A + 1B notes
What changes on first deploy after 2026-04-26
+Bucket name is patronai (not rvdts-fin-processor)

Earlier internal docs referenced rvdts-fin-processor — that's the finance-side bucket. Marauder Scan uses s3://patronai/ exclusively. Set MARAUDER_SCAN_BUCKET=patronai in your environment file.

+RBAC migration runs automatically on first dashboard load

The first request to the dashboard checks s3://patronai/users/users.json. If absent, it bootstraps from your existing ALLOWED_EMAILS + ADMIN_EMAILS env vars one-time. Anyone in ADMIN_EMAILS becomes role=manager + is_admin=true. Anyone else gets role=support.

No env-var changes required. Existing access is preserved exactly. After the migration, edits go through the dashboard's Users settings tab — no redeploy needed for access changes.

+SUPPORT_EMAILS env var is deprecated

Phase 1B's RBAC supersedes the per-role env var. Users get role=support via the Users tab going forward. The SUPPORT_EMAILS var is still read for backward compatibility but no longer needed.

+Re-install agents to pick up Phase 1A finding categories

The new MCP / Agent / Tools / Vector-DB scanners are baked into setup_agent.sh.template at install time. Existing fleet keeps working but won't see new categories until re-installed. Coordinate via Settings → Deploy Agents → Re-issue per recipient.

First scan after re-install runs in baseline mode (wider walk); subsequent 30-min scans run lighter via the first_run.flag mechanism.

+S3 IAM scope additions

The EC2 instance role's S3 policy needs read+write on these new prefixes:

s3://patronai/users/*           # RBAC source of truth
s3://patronai/mcp_hashes/*      # MCP config-change registry
s3://patronai/findings/*        # already exists; new categories land here
+Email module deferred — V1.1 plan in memory

SMTP via AWS Secrets Manager is documented but NOT shipped in V1. Existing SES setup (if any) keeps working untouched. Phase 1B intentionally ships without an email layer; the V1.1 build will replace SES with per-user SMTP credentials stored in Secrets Manager.
