PatronAI's rules engine absorbs the detection engineering — new AI tools (Google Opal, sim.ai, the next thing) get added to the central denylist and pushed to every deployed agent within 6 hours via the Marketplace rules channel.
How rules updates flow
DMG ships once. Rules update silently every 6 hours.
The agent binary is signed and changes only quarterly. Detection rules (the provider denylist and the AI image catalogue) arrive via the PatronAI Marketplace rules channel, sync into your tenant storage, and are cryptographically signed so the agent trusts only authentic updates. Section 5

Your First Day on PatronAI

What you see when PatronAI is live in your company. The Platform Admin and Support Team handle the behind-the-scenes plumbing — you arrive at a working platform.

1
You receive a welcome email
Your Platform Admin invites you into PatronAI. The email contains your sign-in link and role (Security Executive, SecOps Manager, or Support). Click the link to set your password.
2
You confirm your alert email
A separate one-time message asks you to confirm delivery of PatronAI alerts to your inbox. One click. After this, new-AI detections arrive directly in your email as they happen.
3
You sign in to PatronAI Metrics Dash
Your landing page depends on your role. A Security Executive lands on the Exec view — world map, risk heatmap, data-exposure timeline. A SecOps Manager lands on the Manager view — asset inventory, open alerts, pipeline health. Everything is populated with your own environment's data — no empty starter state.
4
You open the PatronAI User Interface
SecOps Managers and the Platform Admin have a second tab — the User Interface — for day-to-day controls. Review the provider denylist, approve internal AI endpoints, trigger a rescan, or export OCSF logs. No commands. No config files. Everything is a button or a form.
5
Alerts begin arriving
Within the first scan cycle (default five minutes), PatronAI begins surfacing unauthorised AI tool usage across your network and source code. Alerts appear in three places at once: the Metrics Dash, your email inbox, and — if wired — your Trinity incident console. You triage from whichever surface you prefer.

Before You See Alerts — a Readiness Checklist

Not a deployment checklist — a readiness one. If all six items below are green, your PatronAI tenant is healthy and ready for you to work in.

Your tenant is live
  • YouWelcome email received and sign-in works
  • YouAlert email address confirmed
  • YouMetrics Dash opens to your role view
  • YouUser Interface opens and saves edits
Your environment is flowing
  • SupportPlatform deployed in your AWS account
  • SupportNetwork telemetry enabled across your VPCs
  • SupportIdentity list uploaded (who owns which device)
  • SupportProvider denylist synced from Marketplace
  • YouApproved internal AI endpoints added to allow list
  • YouPilot device group identified for initial rollout
How to tell PatronAI is working — without logging into anything technical
Open the Metrics Dash. The Pipeline Health tile should show a recent scan timestamp (within the last five minutes) and a green status. The Asset Inventory tile should show a non-zero device count. The Risk Alerts tile should either show today's alerts or an explicit "No new alerts in the last 24 hours" message — never a blank or "no data" state. If any of those three tiles look empty or stale, contact your Support Team.
Section 7

Known Gaps — Documented and Accepted

Four scenarios no network scanner can detect. Disclosed for honest governance reporting.

Gap 1
Fully offline local inference
Model downloaded, data copied via USB, zero network call made. No outbound traffic to intercept. CrowdStrike process monitoring partially mitigates when enabled.
Gap 2
Personal cloud accounts
Personal AWS or GCP account with personal credit card. No corporate IAM footprint. DNS and CrowdStrike network monitoring partially mitigates.
Gap 3
Web UI manual copy-paste
Developer manually copies data into a ChatGPT browser tab. No AI API call made from the corporate network. Browser DLP and clipboard monitoring partially mitigates.
Gap 4 — Accepted risk
Personal laptop outside VPN
Unmanaged device on a home or public network. Out of corporate perimeter entirely. Accepted company risk. Out of scope for this platform.
What closes these gaps
Git scanner (GitLeaks / Trufflehog) in CI/CD catches hardcoded keys at commit time.  ·  Parameter Store enforcement detects personal key usage on authorised domains.  ·  CrowdStrike (when enabled) provides process-level visibility for gaps 1 and 2.  ·  AIRTaaS red team exercises quarterly to test detection coverage.

Phase 1A + 1B — operator additions (2026-04-26)

RBAC — managing users from the dashboard

First-run migration
The first time the dashboard reads `s3://patronai/users/users.json` and finds it absent, it writes a one-time bootstrap from ALLOWED_EMAILS + ADMIN_EMAILS env vars. Anyone in ADMIN_EMAILS becomes role=manager + is_admin=true. Anyone else in ALLOWED_EMAILS becomes role=support. Existing access is preserved without redeploy.
Adding / editing users
Settings → Users tab. Three role pills: exec (purple) / manager (blue) / support (green). Admin checkbox is orthogonal — admins see all views + Settings regardless of base role. Edits go directly to S3; no redeploy needed.
Role visibility matrix
exec sees Exec only. manager sees Manager + Provider Lists. support sees Support + Manager (read-only) + Provider Lists. admin (any base role + checkbox) sees all of the above + Settings; lands on their base role's view by default.

AI Inventory — the new Manager tab

What it shows
Five new finding categories, deduped to ONE row per (owner, device, category, provider) with LATEST observation: MCP servers (Claude Desktop / Cursor / Continue / Cline configs), Workflows (n8n / Flowise / langflow), Scheduled agents (cron / launchd), Tool registrations (`@tool` decorators in repos), Vector DBs (Chroma / FAISS / LanceDB / Qdrant / Milvus).
AI Asset Map — click any owner email
Opens a per-user page with a Plotly Treemap (User → Repo → Category → Asset) plus a nested expander tree below. URL pattern: ?view=asset_map&email=alice@x.com. Click "← back to AI Inventory" to return.
MCP config-change alerts
Server-side hash-flip detection — when the SHA-256 of a user's MCP config differs from the last-known value, the pipeline emits a derived MCP_CONFIG_CHANGED HIGH-severity event. First sighting is silent (no last hash to compare); flips on subsequent scans fire the alert.

Time format — every timestamp now

DD-MMM-YY HH:MM:SS TZ
26-APR-26 14:30:45 IST — rendered in the viewer's local browser timezone (auto-detected via Streamlit 1.32+'s `st.context.timezone`). Hover over any timestamp to see the raw UTC ISO for audit copy-paste. Single helper at dashboard/ui/time_fmt.py; consumed by every grid + chart axis.

Per-grid global search

Search any column
Substring match across all string columns; case-insensitive; live result-count caption. Wired into Risks, Inventory, Logs, AI Inventory, Signals, Rules. Single helper at dashboard/ui/filtered_table.py.

Open-source posture

No SES dependency
The deferred SES email fix is superseded by an optional V1.1 SMTP module using AWS Secrets Manager for per-user credentials. Plan parked in memory; build follows V1 verification. Replaces the hardcoded SES sender path entirely.
Re-install required for Phase 1A
Existing fleet keeps working but won't see the new finding categories until re-installed. The new fragments are baked into setup_agent.sh.template at install time. Coordinate fleet refresh via Settings → Deploy Agents → Re-issue per recipient.
